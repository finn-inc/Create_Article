# kiji — 技術記事作成スキル

以下の4ステップで技術記事を対話的に作成する。各ステップを順番に進め、ユーザーの確認を得てから次へ進む。

---

## STEP 1: 話題の投入（interactive）

### 1-0: URL先行入力チェック

ユーザーの最初のメッセージ（`/kiji` の引数やスキル呼び出し直後の入力）にURLが含まれている場合、そのURLを外部ソースURLとして採用し **1-1 をスキップ** して 1-2 へ進む。

### 1-1: ソースURL収集

    AskUserQuestion(
      question = "記事のソースURLを貼ってください（X投稿、ニュース記事、公式ブログ等）",
      options  = []
    )

### 1-2: 本文・追加情報収集

    AskUserQuestion(
      question = "ソースの本文をコピペしてください。追加情報があれば一緒にどうぞ。\n\n含められる情報:\n・コピペ本文（必須）\n・内部ソース（Slack #timesや夕会の反応など）\n・個人の想い（「ここが技術的にエグい」等の一言メモ）",
      options  = []
    )

**注目アカウント一覧（参考）:**
@OpenAI, @ChatGPTApp, @AnthropicAI, @claudeai, @GoogleAI, @GoogleDeepMind,
@GeminiApp, @googleaidevs, @googleaistudio, @xai

上記を参照元として記録し、記事末尾のソースURL一覧に含める。

---

## STEP 2: 認識の同期とファクトチェック（autonomous → user confirmation）

収集した情報をもとに以下の3点を実施する。

1. **要約**: エンジニア視点で要点を噛み砕いて要約する
2. **AI自身の認識・分析を提示**: 知っていること・知らないことを明示する
3. **信頼性チェック**: ソースの信頼性・矛盾の有無を報告する
4. **推しポイント提案**（個人の想いが未入力の場合のみ）: 記事で推せるポイントを3案提示し、ユーザーに選択してもらう

ユーザーに確認を求める。個人の想いの入力有無でテンプレートを分岐する:

**パターンA（個人の想い入力済み）:**

    AskUserQuestion(
      question = "上記の内容で合っていますか？",
      options  = ["OK、STEP 3へ進む", "修正したい点がある"]
    )

**パターンB（個人の想い未入力 → 推しポイント提案あり）:**

    AskUserQuestion(
      question = "推しポイントを選んでください（記事の核になります）",
      options  = [
        "1: {推しポイント1の要約（50文字以内）}",
        "2: {推しポイント2の要約（50文字以内）}",
        "3: {推しポイント3の要約（50文字以内）}",
        "修正したい点がある"
      ]
    )

※ `{推しポイントNの要約}` は fact-checker の出力から動的に埋め込む。

### サブエージェント実行

Agent ツールで fact-checker を起動し、メインコンテキストを保全する。

    Agent(
      subagent_type = "general-purpose",
      model         = "sonnet",
      description   = "fact-check article sources",
      prompt        = """
        {agents/fact-checker.md の全文を Read で読み込んで埋め込む}

        --- 入力データ ---
        外部ソースURL: {url}
        コピペ本文: {body}
        内部ソース: {internal}
        個人の想い: {thoughts}（未入力の場合は「未入力」と記載）
      """
    )

返却された結果をユーザーに提示し、確認を求める。

---

## STEP 3: 記事構成の確定（autonomous → user confirmation）

### 3-0: 好みの読み込み（autonomous）

`references/preferences.md` が存在する場合、集計サマリを読み込む。
各軸で推奨（★）が付いている選択肢に「(推奨)」ラベルを付けて提示する。
ファイルが存在しない、または履歴が3回未満の場合は推奨ラベルなしで提示する。

**固定ターゲット読者**: 20代後半のミドルWebエンジニア（経験3年以上）。SES・受託開発経験者で、自社プロダクトに関われない不満を持つ。ただし純粋バズ・教養記事の場合はテーマに応じて調整する。

以下の3軸設計で記事構成を提案する。選択肢は固定リストとし、推奨がある場合は `(推奨)` ラベルを付与する:

```
【記事の目的】採用ブランディング / 純粋バズ / 教養・知見共有
【切り口（軸A）】逆説 / 速報 / 代弁 / 発見 / 対比 / 数字の衝撃 / 問いかけ
【展開の武器（軸B）】分類・命名 / 時系列 / 比喩・ゲーム化 / 利害分析 / 事例深掘り / 当事者の声 / 数値シミュレーション
【トーン】熱量型 / 冷静・客観型
【読後の方向感】議論を起こしたい / モヤモヤさせたい / 行動を促したい / 感嘆させたい / 危機感を持たせたい
```

※ 各軸の `(推奨)` は集計サマリの★に基づいて付与する。推奨なしの場合はラベルなし。

AIが最適な構成案を1つテキストで提示し、以下の固定3択で承認を求める:

    AskUserQuestion(
      question = "この構成案でいきますか？",
      options  = ["この構成でOK", "カスタマイズしたい", "全く別の切り口で提案して"]
    )

- 「カスタマイズしたい」→ `options = []`（自由入力）で修正内容を受け取る
- 「全く別の切り口で提案して」→ 異なる切り口・軸Bを使った代替案を提示し、再度同じ3択で確認

加えて、以下のいずれかのテンプレートに従った目次（アウトライン）を提示する。

- **問題提起型**: 問題提示 → 深掘り → 解決策・展望
- **事実積層型**: 事実の羅列 → 含意の解説 → 読者への問いかけ
- **自由形**: テーマに応じて自由に構成する

アウトライン提示後、以下の固定3択で確認する:

    AskUserQuestion(
      question = "このアウトラインで執筆に進みますか？",
      options  = ["OK、執筆へ進む", "アウトラインを修正したい", "構成から練り直したい"]
    )

- 「アウトラインを修正したい」→ `options = []`（自由入力）で修正内容を受け取り、アウトラインを更新して再度同じ3択
- 「構成から練り直したい」→ STEP 3 の冒頭に戻る

ユーザーの承認を得てから STEP 3-9 → STEP 4 へ進む。

### 3-9: 好みの記録（autonomous）

ユーザーが構成を承認した後、選択内容を `references/preferences.md` に自動記録する。

1. ファイル未存在 → 新規作成
2. 「直近の選択履歴」テーブルの先頭に行追加（10行超は末尾を削除）
3. 「集計サマリ」を再計算（各軸の頻度 + 最頻値に★、同率は直近優先）
4. ファイル保存（ユーザー確認不要）

---

## STEP 4: 口調ルールとコンテキストの適用（multi-phase）

### 4-1: Notionから口調資料を取得（autonomous）

`references/notion_sources.md` の「口調・スタイルガイド」に記載された3ページを `notion-fetch` で取得する。

取得失敗時は `references/tone_guide.md` をフォールバックとして使用する。

### 4-2: 模範記事を1〜2本取得（autonomous）

`references/notion_sources.md` の「模範記事」からテーマに近いものを選択し `notion-fetch` で取得する。

### 4-3: 執筆（autonomous）

以下の方針でフルライティングを行う。

- STEP 2 で合意した内容 + STEP 3 の構成に従う
- 口調ルール厳守（常体統一、断定、「のだ」「のである」活用、体言止め禁止）
- STEP 1 の「個人の想い」を強調配置（焼き直し回避の要）
- 文字数: 4000〜6000文字（読了5〜8分）
- 執筆後、重複チェックを実施する

### サブエージェント実行

STEP 4-1・4-2 で取得した口調資料・模範記事をプロンプトに含め、Agent ツールで writer を起動する。

    Agent(
      subagent_type = "general-purpose",
      model         = "sonnet",
      description   = "write full article draft",
      prompt        = """
        {agents/writer.md の全文を Read で読み込んで埋め込む}

        --- 口調資料 ---
        {STEP 4-1 で取得した口調ガイド全文、または tone_guide.md のフォールバック内容}

        --- 模範記事 ---
        {STEP 4-2 で取得した模範記事1〜2本}

        --- 入力データ ---
        ファクトチェック済み情報: {STEP 2 の合意結果}
        記事構成: {STEP 3 の確定アウトライン}
        個人の想い: {thoughts}
        ソースURL: {urls}
      """
    )

返却された記事ドラフトの品質を確認し、問題なければ STEP 4-4 へ進む。

### 4-3.5: 記事本文ファクトチェック（hook自動トリガー）

STEP 4-4 の `notion-create-pages` 実行時に **PreToolUse hook** が自動発火し、初回はNotion出力をブロックする。
ブロック時の指示に従い、以下を実行する:

1. `agents/article-fact-checker.md` を Read で読み込む
2. Agent ツール（model=sonnet）でファクトチェッカーを起動し、記事ドラフト全文を入力として渡す
3. 判定結果に応じて対応する:
   - **PASS** → そのまま再度 `notion-create-pages` を実行する（hookは2回目を許可する）
   - **NEEDS_FIX** → 確信度「高」「中」の指摘箇所を修正し、修正後に再度 `notion-create-pages` を実行する

> **Note:** hookはセッションIDベースのフラグファイル（`/tmp/kiji_factcheck_{session_id}`）でループを防止する。初回ブロック時にフラグを作成し、2回目以降は自動的に許可される。

### 4-4: Notionに出力（autonomous → user confirmation）

`references/notion_sources.md` の「出力先」に記載されたコレクション・プロパティ定義に従い、`notion-create-pages` で記事案データベースに新規ページを作成する。

## エラーハンドリング

| 状況 | 対応 |
|------|------|
| Notion MCP 未接続 | ユーザーに通知し、`articles/YYYY-MM-DD_title.md` にローカルMarkdown出力でフォールバック |
| 口調資料取得失敗 | `references/tone_guide.md` のデフォルトルールを使用 |
| 記事案DB書き込み失敗 | ローカルMarkdown保存 + ユーザーに手動投入を依頼 |
| ファクトチェックhook失敗時 | hookエラーはNotion出力をブロックしない（フェイルオープン）。ログで警告のみ |
